# NanoScope — Changements du 2026-05-17

## 1. Auth HMAC stateless (`backend/auth.py`)

**Problème** : le store de sessions in-memory faisait perdre la session browser à chaque restart du service (fréquent sur Pi).

**Solution** : remplacement par des tokens HMAC signés, stateless.

- Format du token : `user:<unix_ts>:<hmac-sha256>`
- Secret : SHA-256 de `NANOSCOPE_PASSWORD_HASH` — stable entre restarts, invalidé automatiquement si le mot de passe change, aucune config supplémentaire
- Expiry : 30 jours, vérifiée à chaque requête
- Logout : efface le cookie côté client uniquement (suffisant pour single-user)
- Les sessions survivent désormais aux restarts du service

## 2. WS push d'invalidations TanStack Query

**Problème** : SessionDetail pollingait 5 endpoints toutes les 5 s de façon aveugle. Plus généralement, toutes les pages pollingaient en continu sans raison (10–30 s).

**Solution** : le backend surveille les mtimes des fichiers de session et pousse des invalidations ciblées via le WebSocket existant.

### Backend

**Nouveau fichier** : `backend/services/session_watcher.py`
- Scanne `data/v2-sessions/ag-*/sess-*/` toutes les 2 s
- Retourne `{(ag_id, sess_id): max_mtime}` en lisant les mtimes de `inbound.db`, `outbound.db`, `.heartbeat`

**Modifié** : `backend/main.py` — boucle WS
- Tick 2 s : compare les mtimes, émet `{ type: "invalidate", agentId, sessionId }` sur tout changement
- Tick 10 s (inchangé) : émet `{ type: "metrics", data: {...} }`

### Frontend

**Modifié** : `frontend/src/hooks/useWebSocket.ts`
- Appelle `useQueryClient()` depuis le hook
- Sur `type: "invalidate"` : invalide 8 clés TanStack Query :
  - `["agents"]` — compteurs de sessions dans le dashboard
  - `["messages"]` — page Communications
  - `["sessions", agentId]` — liste des sessions d'un agent
  - `["session-detail", agentId, sessionId]`
  - `["session-queue", agentId, sessionId]`
  - `["session-delivery", agentId, sessionId]`
  - `["session-conversation", agentId, sessionId]`
  - `["session-logs", agentId, sessionId]`

**Modifié** : `frontend/src/hooks/useApi.ts` — polling réduit à du fallback pur

| Hook | Avant | Après |
|------|-------|-------|
| `useMetrics` | 30 s | 60 s + `refetchIntervalInBackground: false` |
| `useAgents` | 15 s | 60 s + `refetchIntervalInBackground: false` |
| `useMessages` | 10 s | 60 s + `refetchIntervalInBackground: false` |
| `useSessions` | 10 s | 60 s + `refetchIntervalInBackground: false` |
| 5 hooks `useSession*` | 5 s | 30 s + `refetchIntervalInBackground: false` |

Le WS gère toutes les mises à jour en temps réel (latence ≤ 2 s). Le polling n'est qu'un filet de sécurité si le WS tombe, et s'arrête automatiquement quand l'onglet passe en arrière-plan.
